## Console

Split by cause (per the console policy — real `jsErrors` fail; environment artifacts and error-path `networkErrors` do not):

**Environment artifact (NOT a FAIL) — extension-injected, not app code:**

1. React hydration mismatch on `/sign-up`, triggered on initial page load (step 01, before any interaction):
   ```
   A tree hydrated but some attributes of the server rendered HTML didn't match the client properties. This won't be patched up. This can happen if a SSR-ed Client Component used:

   - A server/client branch `if (typeof window !== 'undefined')`.
   - Variable input such as `Date.now()` or `Math.random()` which changes each time it's called.
   - Date formatting in a user's locale which doesn't match the server.
   - External changing data without sending a snapshot of it along with the HTML.
   - Invalid HTML tag nesting.

   It can also happen if the client has a browser extension installed which messes with the HTML before React loaded.

   %s%s https://react.dev/link/hydration-mismatch

     ...
       <FormField control={{...}} name="name" render={function render}>
         ...
           <Input autoComplete="name" placeholder="Ada Lovelace" data-testid="signup-name" name="name" ...>
             <input
               ...
               aria-invalid={false}
   -           style={{caret-color:"transparent"}}
             >
   ```
   The diff shows a `style={{caret-color:"transparent"}}` attribute on the client-rendered `<input>` for every signup field, absent from the server-rendered markup. **Traced to source (follow-up investigation):** `caret-color` appears nowhere in `app/` (grepped `input.tsx`, `form.tsx`, `signup-form.tsx`, `root.tsx`, entry files, `app.css`). React serializes inline styles into SSR HTML, so a client-only style means it was written to the DOM after render by a browser extension (password-manager caret hiding). The committed `playwright.config.ts` uses clean Desktop Chrome and will not reproduce this. → **environment artifact, not an app bug.** The `feature-verifier` now launches with `--disable-extensions` + a clean profile so this cannot recur.

**networkErrors (expected — not a FAIL):**

2. `401@E1` during the bad-login attempt:

   ```
   Failed to load resource: the server responded with a status of 401 (Unauthorized)
   ```
   Expected HTTP status for a rejected sign-in (how `authClient.signIn.email` surfaces the error the UI renders as `login-error`). Fired on an error-path step → does not count against the verdict.

## Findings for main thread

- **Not a bug (resolved)**: the `/sign-up` hydration warning is a browser-extension artifact (`caret-color: transparent` injected onto inputs), confirmed absent from `app/` by grep. No code change. Verifier hardened with `--disable-extensions` + clean profile so it can't recur.
- **Expected, not a bug**: the 401 during the bad-login attempt is the natural network response for a rejected sign-in (an error-path `networkError`).
- **Gotcha for future verifiers of this feature**: `/login` and `/sign-up` both redirect an authenticated session to `/dashboard` in their loaders (`app/routes/authentication/login.tsx:12`, `app/routes/authentication/sign-up.tsx:12`). Any script that signs up/in and then wants to re-visit `/login` or `/sign-up` in the same run must do so in a fresh `browser.newContext()` (no cookies), not the same authenticated page/context.
- No missing `data-testid`s — all selectors given (`signup-name`, `signup-email`, `signup-password`, `signup-confirm-password`, `signup-submit`, `dashboard-account`, `login-email`, `login-password`, `login-submit`, `login-error`) matched real, confirmed markup in `app/routes/authentication/components/signup-form.tsx`, `login-form.tsx`, and `app/routes/dashboard/_index.tsx`.

## Verdict rationale

Both the golden path (fresh signup → `/dashboard`) and the error path (bad credentials → inline `login-error`, still on `/login`) worked exactly as specified — every step resolved, every assertion held, and all three screenshots were captured. **Verdict: PASS.** The one console warning (a `/sign-up` hydration mismatch) was investigated and traced to a browser-extension DOM injection (`caret-color: transparent`, absent from `app/`), not an app defect — the committed clean-profile Playwright config does not reproduce it, and the verifier now launches extension-free. The 401 during the bad-login step is an expected `networkError`.
